请在当前项目的cvulnweb 目录下，编写一套基于libmicrohttpd的C Web测试代码，用于安全代码分析工具、漏洞检测工具的深度能力验证。由于C语言没有原生的面向对象特性，需通过结构体和函数指针模拟类、继承、接口等概念。整体遵循分目录、分文件原则，结构规范、注释清晰、代码简洁无冗余。

一、整体目录与文件拆分规则
1. 公共父结构体、通用子结构体定义统一放置在 common 目录；
2. 接口定义（函数指针结构体）单独存放至 interface 目录，接口与实现拆分为独立文件；
3. 按照漏洞类型划分独立子目录：cmd_exec（命令执行）、code_exec（代码执行）、ssrf（SSRF漏洞）；
4. 每个漏洞目录下区分两类测试文件：纯结构体继承调用文件、接口实现类调用文件；
5. 根目录保留总入口文件 main.c、通用跨文件函数调用文件 func_call.c。

二、代码功能与场景要求
1. 漏洞类型要求
   集成命令执行、代码执行、SSRF三类高危漏洞，同时包含两种数据源场景：
   - 外部可控数据源：通过 HTTP 请求参数接收前端传入参数，构造完整漏洞调用链；
   - 硬编码无外部数据源：内置固定参数的漏洞代码；
   所有漏洞点嵌入到父结构体方法中，通过「结构体包含、函数指针模拟多态、跨文件调用」形成多层深层调用链。

2. 模拟面向对象调用场景（全覆盖）
   - 基础调用：同一文件内函数/结构体调用、跨文件函数调用、跨文件结构体实例与方法调用；
   - 结构体组合模拟继承：编写独立父结构体 VulnParent、独立子结构体 VulnChild（包含VulnParent作为成员），实现子结构体调用父结构体漏洞方法的链路；
   - 函数指针模拟接口：独立定义接口结构体（包含函数指针），每个漏洞目录编写对应的实现，实现「接口 + 结构体」组合调用；
   - 多层调用链：构造 外部请求参数 → 调用入口 → 子结构体/接口实现 → 父结构体 → 漏洞点 的完整长调用链路。

3. 代码规范要求
   1. 每个文件功能单一、职责清晰，漏洞位置、调用关系添加明确注释标注；
   2. 所有文件路径引用合理，头文件包含无路径错误；
   3. 总入口页面整理所有测试链接，方便直接访问各个测试用例；
   4. 保留纯结构体继承调用测试用例、接口实现类测试用例两类场景，分开独立测试。

三、输出要求
1. 先输出完整的目录树结构；
2. 再按目录顺序，依次输出每一个文件的完整可运行C代码；
3. 代码保证可直接编译运行，无需额外配置。
